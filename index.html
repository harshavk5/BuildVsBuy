Section 01

Executive Summary

This document provides a structured assessment of the viable open-source replacement options for BMC Helix Operations Management (BHOM) monitoring estate. The review is scoped to Unix hosted infrastructure and explicitly treats Kubernetes as a future-state concern, not a current requirement.

Primary Objective
Evaluate and find replacement for functional capability of BMC Patrol and BHOM across a 60,000+ Unix/Linux and database estate, with primary migration scope focused on 43,000 production/contingency servers, while preserving the production support team's ability to manage monitoring without architectural complexity.

Two tools have been evaluated in depth: Prometheus + Grafana (PAG stack) and Zabbix. Both are open-source, Unix-hosted, and commercially supportable. They differ fundamentally in architectural philosophy, operational model, and suitability with the current monitoring culture.

The selection criteria is not purely technical. The operating model, team capability, and production support ownership model are equal determinants. Both are addressed explicitly in the analysis tabs.

Decision Parameters — Confirmed
  • Primary lens: Reduce operational risk — prefer the efficient and high performance solutions that preserves existing monitoring workflows and minimize digital footprint.
  • Platform ownership: Shared model — production support team for day-to-day operations, supported by engineering team. The platform must be operable by BMC-trained engineers for BAU. The engineering team handles architecture, upgrades, and complex troubleshooting. This slightly reduces operational risk compared to prod-support-only ownership, but does not change the fundamental requirement: BAU monitoring operations must not require platform engineering skills.
  • Team familiarity: BMC-trained only. Zero prior exposure to Prometheus/Grafana or Zabbix. Training investment is a programme cost, not a background assumption.
  • AIX / Solaris: Being phased out. Not a monitoring priority. Excluded from migration scope — management-approved out of scope for this programme.
  • Self-service portal for custom monitoring: Direction is a management decision. Three paths quantified in Section 07 (portal on Day 1 / portal deferred / portal retired in favour of GitOps). Operational load profile differs materially across the three. Platform recommendation assumes Option 1 or Option 2.
  • Blackout portal: Separate portal for app-team-driven alert suppression and blackout windows (Section 06). Two distinct modes in production use — suppress (all alerts cancelled automatically during window) and blackout (alerts still above threshold on next poll after blackout window end get forwarded). The replacement platform must reproduce both scenarios. Native on Zabbix (two maintenance modes); composite on the Prometheus stack.
  • Kubernetes: Management decision not made. Uncertain timeline. Treated as future-state and must not constrain platform selection for the current programme, but the selected platform must not actively block K8s adoption when it arrives.

Section 03

Current Monitoring Model

Understanding the current BMC Patrol / BHOM model is the prerequisite for any accurate platform comparison. The following describes the functional model that must be preserved or replaced.

Baseline Policy Model

Policies are tag-driven. A server tagged LINUX automatically receives the baseline configuration covering CPU, memory, uptime, kernel status, filesystem, process monitoring, and agent health. This is a push-down model, the BHOM server applies the policy to agents at registration or tag assignment. No per-server configuration is required for baseline coverage. Each Database has it's own Tag based monitoring.

Custom Policy Model

Per-server custom policies are managed through a self-service portal. Application teams submit requests specifying log file paths, process names, filesystem mount points with custom alerting thresholds, and scripts to be executed. The onboarding team configure these requests through Patrol KM configurations and deploys them via BHOM Monitor/Alarm Policies. Approximately 200 such changes occur per month.

Script Monitoring

Scripts are monitored via two Patrol KM patterns:

  • Scripting KM: ~700 servers. A Patrol KM executes a script, evaluates its output (numeric return code or text), and raises an alert based on threshold or regex match. This is a native Patrol capability — the agent executes and evaluates the script within its own runtime.
  • Log KM (script output to log): Several hundred more servers. A script writes its output to a log file. The Patrol log KM tails the file and fires alerts on pattern matches. This is a hybrid model — the script is independent; Patrol reads its output.

Log Monitoring

The Patrol Log Management KM monitors log files per-host with configurable regex patterns, rotation handling, and alert conditions. Per-server log paths, patterns, and thresholds are the most common type of custom policy request.

Ping / Reachability

ICMP-based ping monitoring is managed through Patrol. All in-scope servers are pinged at regular intervals. Alert on failure or timeout.

Agent Disconnect Detection

When a Patrol agent stops communicating with BHOM, an alert is raised natively. No additional configuration is required. This is first-class functionality in the current platform.

Event Routing

All alerts pass through a separate GMOM / event management layer that handles enrichment (adding CI data, environment, priority), deduplication, suppression, and routing to ServiceNow for incident creation. The replacement platform must forward structured events to this layer — it does not replace GMOM.

Section 05

Current Pain Points with BMC Patrol / BHOM

TSOM → BHOM Architecture Migration — Where We Came From, Where We Are

The current state is the result of an in-flight architecture migration from BMC TrueSight Operations Management (TSOM) — the legacy on-premises product line — to BMC Helix Operations Management (BHOM), the SaaS-delivered successor. Both products share the same Patrol agent and KM model on the end-point side, but the hosting architecture is materially different. Understanding what we have already retired is the prerequisite for understanding what the replacement platform must absorb.

Progress Achieved on the BHOM Migration
  • Server footprint reduced from ~200 to ~30 for the 43k production estate. Efforts spent on pPatching, vulnerability remediation, and OS maintenance for the retired ~170 servers no longer required.
  • Central control ownership transferred to BMC via the SaaS model. We no longer own Presentation Server, Infrastructure Management Server, or repository tier patching and CVE response.
  • The TSOM Integration Services aggregation tier is retired. BHOM's architecture removes the IS layer — PATROL Agents connect directly to BHOM SaaS via a BlueCoat forward-proxy tunnel. The ~750-agent-per-IS ceiling that was the dominant TSOM scaling pain point is no longer structurally present.
  • The remaining ~30 servers are the small number of ICMP ping / availability hosts for targets that cannot be probed from BMC's SaaS infrastructure.

Active Pain Points — What the Replacement Programme Must Solve

With the TSOM IS bottleneck retired by the BHOM migration, the remaining pain points sit in three areas: edge-tier capacity (BlueCoat proxies and agent placement), the unchanged PATROL/KM operational model on the host, and internal components surrounding the platform.

  • BlueCoat forward-proxy capacity is the new edge constraint. Each BlueCoat proxy has a finite throughput and concurrent-connection ceiling for tunnelled HTTPS sessions to BHOM SaaS. Capacity planning has moved from "agents per IS" to "agents per BlueCoat proxy". Tunable, but must be designed. A misplaced BlueCoat proxy still creates a blast radius for the agents behind it.
  • Single BlueCoat proxy per region — no agent-side failover. The current deployment runs one BlueCoat proxy per region with no redundancy configured at the agent or platform level. If the proxy becomes unavailable, all PATROL Agent visibility for that region is lost until recovery and BHOM cannot receive data, cannot detect agent disconnect, and cannot evaluate alerts. This is an unaddressed single point of failure in the current BHOM edge architecture. The network team owns the BlueCoat appliance; the monitoring team's scope is ICMP ping monitoring only — patch, vulnerability. The replacement platform must not replicate this availability gap.
  • PATROL Agent placement and density on hosts is unchanged from TSOM. The agent itself, the KM versions running inside it, and the per-host configuration burden are identical. The BHOM migration led to retiring HSBC managed central tier and the aggregation tier components but did not change the configuration efforts and operating model of the end points.
  • KM dependency complexity: PATROL Knowledge Modules are versioned independently from the agent and the BHOM platform.
  • Scripting KM operability: ~700 servers depend on PATROL Scripting KM with internal scripts. Lifecycle ownership of those scripts (placement, version, error handling, output schema) is split between application teams and onboarding — a documented friction point unrelated to which platform sits behind it.
  • Self-service portal is custom-built: Not native to BMC, we maintain this portal for ~200+/month policy changes. Its existence (or absence) in the replacement is a separate management decision; see Section 07.
  • Maintenance blackout / suppression portal is custom-built: A separate portal handles app-team-driven alert suppression and blackout windows during DB upgrades, patching, and other planned activity. Its semantics (suppress vs blackout, with post-window threshold re-evaluation) must be replicable on the replacement platform — see Section 06.
What This Implies for the Replacement Platform

The BHOM migration has already proved that reducing the server footprint is the highest-leverage operational improvement available — ~85% reduction in maintenance load with no change to the BAU operating model for application or onboarding teams. The replacement platform must protect that gain and must not reintroduce the architectural bottleneck that BHOM retired.

  • SaaS is not a requirement. We have already absorbed the regulatory, network, and data-residency tradeoffs of a SaaS monitoring platform and is comfortable with either direction. The replacement solution does not need to be SaaS to be acceptable.
  • Low production support footprint is a hard requirement. Whatever platform is selected must minimise the number of servers that production support must patch, scan, and certify. A self-hosted platform is acceptable only if the total server footprint stays in the same order of magnitude as today's BHOM-era ~30 servers — not the legacy TSOM-era ~200.
  • Must not reintroduce a hard per-collector agent ceiling. TSOM's ~750-agent-per-IS limit was the single most painful historical constraint; the BHOM architecture retired it and the replacement solution should not reintroduce the hard per-collector agent ceiling. Horizontally-shardable collection (Prometheus) or tunable proxy density (Zabbix) both satisfy this — a fixed-density aggregation tier does not.
  • Agent-to-backend path must be tunnel-friendly. The current estate already traverses a BlueCoat forward-proxy tier. Replacement platforms must either support that pattern natively or support a functional equivalent (regional proxies, scrape gateways) without requiring direct agent→central-DB connectivity across security zones.
  • Must natively support (or cleanly compose) the maintenance blackout / suppress semantics currently provided by separate dedicated portal — see Section 06. This is a user-facing capability that cannot be downgraded during migration.
  • Vendor support model must match a tier-1 environment regardless of SaaS vs self-hosted delivery.
Section 06

Maintenance Blackout / Suppression Portal — Current Behaviour and Replacement Requirements

Separate from the policy self-service portal (Section 07), a dedicated blackout/suppress portal is provided for application teams. Its purpose is to give app teams direct, self-service control over alert noise during planned maintenance — DB upgrades, patching cycles, application releases, and any other window where alerts would otherwise be expected and unhelpful. This portal is sits in front of BHOM, and is part of the BAU operating model. The replacement programme must reproduce its semantics — not just "silence alerts", but the specific two-mode behaviour described below.

Two Distinct Modes — Both Are In Production Use
  • Suppress mode — During the configured window, every alert triggered on the selected hosts is suppressed by default. Nothing is forwarded to event management. The window starts, alerts go quiet, the window ends, normal alerting resumes. Used for activities where the team accepts that any alert during the window is expected noise and does not require triage.
  • Blackout mode — Alerts triggered during the window are also held back during the window, but with a critical difference: any alert that is still above its configured threshold on the next polling cycle after the window ends is forwarded for further processing as a real event. Used for activities where the team wants noise suppressed during the work but wants a guaranteed signal if the system has been left in a bad state when the window closes.

Why the Two-Mode Distinction Matters

Suppress is "I know it will be noisy, ignore it". Blackout is "I know it will be noisy during, but tell me if it's still broken after". The two are not interchangeable and substituting one for the other introduces real operational risk. A window-end re-evaluation is the difference between "the DB upgrade went fine" and "the DB came up degraded and nobody knew until the next page". App teams pick the mode based on the risk profile of the maintenance activity. The replacement platform must support both modes with this exact semantic — a generic "silence everything" capability is functionally insufficient.

Operating Model

  • Self-service: App teams create blackout/suppress windows directly via the portal. Onboarding and Production Support do not gate these requests — that is the entire point of the portal.
  • Scope: Per-host or per-host-group. Mode (suppress vs blackout) selected at window creation.
  • Volume: Active throughout BAU — every patching cycle, every DB upgrade, every release window across the estate generates entries. Independent of and additional to the ~200+/month policy change traffic in Section 07.
  • Audit: Portal records the details of the requestor, target server/objects, suppression/blackout window and mode selected, used for incident retrospectives and change-management reconciliation.

Replacement Platform Requirements

  • API-driven window creation/cancellation for the portal backend. The portal is staying — only its backend changes when the platform changes.
  • Per-host and per-host-group scoping with configurable start/end timestamps.
  • Mode selection at window creation mapping to the two semantics above. If the platform offers only one mode natively, the second must be reproducible via composition (rule logic, post-window evaluation, or portal-side enforcement) and the engineering effort to do so must be estimated in the platform decision.
  • Window-end re-evaluation on the next polling cycle for blackout mode, with forwarding to GMOM only when the underlying condition is still above threshold.
  • Audit trail and lifecycle visibility — windows are queryable, cancellable, and reportable.

Decision Implication
The blackout/suppress portal is a self-service, BAU-critical capability that is already operational. Zabbix maps both modes to native maintenance behaviours with no glue code — the portal is a backend swap. Prometheus maps suppress cleanly via Alertmanager silences but requires internal logic to reproduce blackout's post-window re-evaluation semantic. This is consistent with the broader pattern in the Tool Comparison tab: Zabbix native where Prometheus needs composition. The blackout requirement does not change the platform recommendation — but it adds another costed engineering workstream to the Prometheus path.
Section 07

Self-Service Portal — Management Decision Required

The self-service portal is non-BMC component that sits in front of the monitoring platform. It is used by application teams to request monitoring configuration changes and by the onboarding team to translate those requests into platform configuration. Whether the replacement programme includes a portal or defers it, or decommissions the concept entirely is a management decision, not a technical one. This section quantifies the operational load in each direction so the decision can be made on real numbers.

Framing
This section does not recommend. It describes what the Production Support and Onboarding teams absorb if the portal does not exist — for both existing-server configuration changes and new-server build configurations. Management owns the direction.

Today's Baseline — With Portal

The portal today handles approximately 200+ policy change requests per month across the estate. Each request covers one of: a new log monitoring rule, a process check, a filesystem threshold change or a bulk onboarding event for a new server. The portal abstracts the platform API — app teams submit a structured request and the portal applies it to the monitoring platform.

  • App team effort per request: 5–10 minutes (portal form submission)
  • Production support team effort: near-zero for routine changes — the portal is self-healing for typical requests
  • Change traceability: full — every change is timestamped, attributed, and logged by the portal

Scenario A — No Portal at Cutover (Operational Load Analysis)

If the replacement programme proceeds without a portal, the same 200+/month change workload must be absorbed by manual engineering effort against the new platform's API or UI. The following estimates are conservative, based on the same request mix and volume.

Secondary Costs — Not Just Hours
  • Change traceability degrades. Without portal-driven logging, change attribution falls back to platform-native audit (Zabbix audit log or Git commit history). Workable but requires discipline and does not automatically link a change to the requesting app team.
  • App team friction increases. App teams currently self-serve. Without a portal, every monitoring change becomes an onboarding ticket — ticket queue depth and mean time to fulfilment will increase visibly.
  • Custom configuration becomes a bottleneck. 200+ tickets/month against a onboarding queue is a resource funding conversation — either more onboarding headcount or accepted SLA degradation for monitoring changes.
  • Error rate increases. Manual changes, even against a clean API, have a measurable error rate. A portal with validation reduces this — manual work reintroduces it.
  • Onboarding of new servers becomes a manual coordination exercise with the server build pipeline. Today the portal is the handoff point; without it, the handoff moves to email, tickets, or a new process that must be designed.

Scenario B — Portal Deferred to Post-Cutover

If management accepts a portal gap during the transition — with the intent to build the portal after the platform is live — the load above applies only during the gap period. Realistic gap duration: 6–12 months (portal build + pilot). During that window, the Onboarding and Production Support teams absorb ~1.7–2.6 FTE of sustained manual work, and app team friction/ticket volume is visibly elevated. Once the portal lands, the baseline returns to near-zero marginal load.

This path is feasible but must be resourced. The commitment is a clear, time-bound temporary FTE uplift for the Onboarding team — not a permanent operating model change.

Scenario C — Portal Replaced Permanently by GitOps or Direct API

If management decides the portal concept is retired and app teams submit monitoring changes directly via Git pull requests or a thin API surface: the onboarding team role shifts from applying changes to reviewing PRs. Net effort is comparable to today's portal-driven model if app teams are trained and tooling is good; higher if not. This path is a cultural/operating model change for the application estate, not purely a monitoring decision.

Management Decision
  • Option 1 — Portal on Day 1 (today's model preserved): Fund portal development in Phase 1. Zero operating-model change for app and onboarding teams. ~0.5–1.0 FTE of portal engineering cost during Phase 1, near-zero BAU impact after.
  • Option 2 — Portal deferred (accept gap): Accept ~1.7–2.6 FTE temporary uplift on Onboarding/Production Support for 6–12 months. Portal builds post-cutover. Lower upfront engineering cost; higher BAU cost during the gap.
  • Option 3 — Portal retired: Permanent operating model change. Requires training and tooling investment.

The platform selection below assumes Option 1 or Option 2. Option 3 shifts the conversation materially and should trigger a re-review. The solution is not limited to the options given above, this will be a management call.

Prometheus Analysis — 01

Architecture Overview

Prometheus is a pull-based metrics collection and alerting system, originally developed at SoundCloud and now a CNCF-graduated project. It stores time-series metrics in its own TSDB, evaluates alerting rules in PromQL, and routes alerts via Alertmanager. It does not natively handle logs. Log collection is a separate concern handled by Loki (Grafana's log aggregation system). Grafana provides the visualisation layer.

At this scale, Prometheus cannot be deployed as a single instance. The required architecture is sharded and federated.

Collection Layer — Per Host
  • Grafana Alloy — unified agent. Handles log tailing to Loki, local exporter scrape relay, and script wrapper invocation. Replaces the Patrol agent role for log and metric forwarding.
  • Node Exporter — OS-level metrics. CPU, memory, disk, filesystem, kernel, network. Exposed on port 9100. Scraped by regional Prometheus shards.
  • Textfile Collector — directory-based metric file ingestion. Scripts write .prom files; Node Exporter reads and exposes them. Core mechanism for script monitoring.
  • DB Exporters — oracledb_exporter, postgres_exporter, mongodb_exporter for DB instances. Sybase and DB2 databases doesn't have any native exporters, hence cannot be monitored.
  • Blackbox Exporter — ICMP probe for ping monitoring. Deployed per-region, not per-host.
Regional Aggregation Layer — Per Region (UK, US, HK, MX)
  • Prometheus Cluster (sharded): 3–5 Prometheus instances per region. Shard by function: OS metrics shard, DB metrics shard, custom exporter shard. Each shard runs as an active-active pair for HA (Prometheus has no native clustering — HA via duplication).
  • Loki Cluster: Distributed log ingestion. Alloy agents on each host forward logs to regional Loki. Log-based alerting rules replace Patrol Log KM alert conditions.
  • Alertmanager (2-node mesh): Active-active HA. Routes to GMOM webhook, ServiceNow API, email, on-call tool.
Global Federation Layer
  • Thanos or VictoriaMetrics Cluster: Long-term retention, cross-region query federation. Thanos has broader enterprise adoption; VictoriaMetrics has lower operational complexity. Decision should be made based on team capability in Phase 1.
  • Object Storage (S3-compatible): Long-term metric block storage. On-prem: Ceph or MinIO. Enables 12–24 month retention without local disk growth on Prometheus nodes.
  • Grafana Enterprise: Global unified dashboarding and RBAC. Self-service portal equivalent via Grafana's API and role model.
Policy Governance Layer
  • Git Repository: Prometheus alerting rules, recording rules, Loki alerting rules, Alloy configs, scrape configs — all stored as version-controlled YAML files.
  • CI Pipeline: Validates and deploys rule changes. PR/MR review process provides audit trail for every policy change.
  • Ansible: Agent deployment, exporter installation, config distribution at 60k server scale.

Prometheus Analysis — 03

Script Monitoring in Detail

This is the functional area where Prometheus's distance from the current BMC model is greatest, and where it is most commonly understated in platform evaluations. This section does not understate it.

The BMC Patrol Scripting KM Model

The Patrol agent executes a script at a configured interval, captures its output (stdout, return code, or a structured format), evaluates it against threshold rules, and raises an alert within the agent's own runtime. The script may be shell, Perl, Python, or any executable. Output evaluation is built in — no external tooling required.

Pattern 1: Textfile Collector (Recommended for Most Scripts)

A script runs via cron or systemd timer. It writes a .prom formatted file to /var/lib/node_exporter/textfile/. Node Exporter reads the directory on each scrape and exposes the metrics. Prometheus evaluates them against alerting rules.

  • Decoupled: script execution is independent from monitoring. Script can fail without blocking Node Exporter.
  • Simple: the wrapper is 5–15 lines of shell to convert script output to Prometheus text format.
  • Limitations: execution frequency is bounded by the cron/timer interval, not by Prometheus scrape interval. No sub-minute execution without systemd timer configuration.
  • Staleness risk: if a script stops running, the .prom file becomes stale. Must implement a file timestamp check to detect silent failures.

Pattern 2: Script Exporter (For Synchronous Evaluation)

The prometheus-community/script_exporter executes scripts at scrape time, synchronously. Closer to the Patrol scripting KM model. Returns script output as Prometheus metrics.

  • Execution is on-demand at Prometheus scrape time — consistent with the Patrol polling model.
  • Synchronous execution risk: if 60,000 hosts each have a 30-second script executing at scrape time, Prometheus scrape timeouts become a design concern.
  • Must be deployed as an additional service per host. Adds operational footprint.
  • Not recommended as the default pattern for all 700 scripts. Should be used selectively for scripts requiring real-time evaluation.

Migration Effort for 700 Scripting KM Equivalents

Realistic effort assessment

The 700 scripts are not uniform. Some will be simple numeric return code checks — 15 minutes of wrapper development each. Others will be complex multi-value output scripts with internal state or Patrol-API interaction — these require full refactoring. A realistic estimate for the scripting KM migration alone:

  • ~60% simple wrappers: 2–4 hours each including test and deployment
  • ~30% medium complexity: 1–2 days each
  • ~10% complex/Patrol-API-coupled: 3–5 days each or decommission

Total engineering effort: 500–900 person-days for scripting KM migration alone. This must be scoped and resourced explicitly.

Prometheus Analysis — 04

Log Monitoring in Detail

Component Chain Required

Log monitoring in the Prometheus/Grafana stack is not a single component — it is a pipeline: Alloy → Loki → Loki Ruler → Alertmanager. Each step must be deployed, configured, and maintained.

  • Grafana Alloy on each host: configured with a loki.source.file block specifying log file paths and labels. Per-host custom log paths require per-host Alloy config.
  • Loki Cluster: receives structured log streams from Alloy agents across all hosts. Must be sized for the log ingestion volume of 43k servers.
  • Loki Ruler: evaluates LogQL alerting rules on log streams. Fires alert payloads to Alertmanager when rules match.
  • Alertmanager: routes log-based alerts to GMOM alongside metric-based alerts.

Per-Host Log Path Management

The current BMC model allows the portal to specify a log file path per server and a regex pattern. In the Prometheus/Grafana model, this translates to a per-host Alloy configuration block specifying the log path and a per-alert-rule LogQL filter. These must be managed as code (in Git) and deployed via Ansible or CI pipeline.

This is a more complex operational model than a Zabbix log item. The complexity is manageable but must be acknowledged: changing a log path for a server is a Git commit, CI run, and Ansible task — not a single API call.

Real-Time Alert Latency

The default Loki Ruler evaluation interval is 1 minute. Sub-minute alerting on log events is technically achievable by reducing the evaluation interval, but at the cost of increased Loki Ruler and Loki query load. For a environment with large log volumes, the tradeoff must be explicitly sized and tested. In Patrol, log KM evaluation is near-real-time (seconds). If sub-minute log alerting is a hard requirement, this is an area where Prometheus/Loki requires careful validation.

Prometheus Analysis — 06

Operational Fit for the Production Support Team

Ownership Model — Shared, But Not Sufficient for Prometheus
Platform ownership is a shared model: production support for BAU, a small platform team for architecture and complex operations. This is better than prod-support-only — the platform team can own Prometheus infrastructure, upgrades, and PromQL/LogQL rule management. However, it does not resolve the core risk: the production support team, who handle alert acknowledgement, policy changes (200+/month), and day-to-day monitoring operations, are BMC-trained only. They will interact with Grafana, Alertmanager, and the portal daily. When the portal is unavailable, they will need to diagnose the underlying component pipeline (Alloy → Loki → Ruler → Alertmanager). When an alert is missing, they will need to triage across Prometheus scrape state, Loki ingestion, and rule evaluation. These are Prometheus-native diagnostic skills, not BMC-transferable ones. The small platform team cannot be the escalation path for every BAU incident. Prometheus's operational model introduces a daily friction cost for the production support team that Zabbix does not.

Day-to-Day Operations

  • Alert acknowledgement and silencing: Alertmanager UI or Grafana OnCall. Manageable. Similar concept to BMC suppress/acknowledge.
  • Adding a new log monitoring rule: In BMC, an API call from the portal. In Prometheus: edit a Loki alert rule YAML in Git, raise a PR, CI validates, Ansible deploys. Significantly more steps unless the portal abstracts this.
  • Changing a filesystem threshold: In BMC, update a policy parameter via API. In Prometheus: update the alert rule condition or create a per-host recording rule. More verbose but automatable.
  • Diagnosing a missing alert: In BMC, check policy deployment status in BHOM console. In Prometheus: query Prometheus targets, check Alloy logs, check Loki ingestion. Requires familiarity with multiple components.
  • Cardinality management: No equivalent concept in BMC. A new operational skill the team must develop. Undisciplined label usage at 43k scale will cause Prometheus OOM incidents.

Training Requirement

With a shared ownership model, training splits across two tracks:

  • Platform team (2–4 engineers): Full Prometheus/Grafana/Loki training — PromQL, LogQL, Alertmanager routing, sharding architecture, cardinality management, GitOps pipeline. 3–4 months to operational competence from a BMC-only background. These engineers own the infrastructure, rules management, and escalation.
  • Production support team (BAU operators): Grafana dashboards and alert acknowledgement (2–3 weeks). Alertmanager silencing and routing (1 week). Portal usage for policy changes (1 week if portal properly abstracts the complexity). They must not need to understand PromQL, Loki rules, or Git for normal BAU operations — the portal and platform team must insulate them from this. If that insulation fails at any point, operational risk increases immediately.

The shared model makes Prometheus more viable than prod-support-only ownership. It does not make it equal to Zabbix for the BAU operator experience.

Prometheus Analysis — 07

Patching and Day-2 Operations

Agent-Side (Alloy / Node Exporter)

Both Grafana Alloy and Node Exporter are statically compiled Linux binaries. They have no runtime dependencies beyond the OS. OS patching and rebooting does not require agent reconfiguration — systemd restarts the agent service automatically. Agent binary upgrades can be orchestrated via Ansible on the 15-day patch cycle. This is operationally simpler than Patrol KM patching.

Platform-Side (Prometheus, Loki, Alertmanager)

All components are long-running server processes managed by systemd. HA design (active-active pairs or shards) allows rolling upgrades: patch one instance, validate, patch the next. A Prometheus scrape gap during restart is approximately one scrape interval (60 seconds by default) — acceptable for operational monitoring. Loki and Thanos/VictoriaMetrics use object storage; component restarts do not lose data.

OS Compatibility

All components run on RHEL 7+, Oracle Linux 7+, Debian, and Ubuntu. RHEL 6 hosts are not supported by current Grafana Alloy or Prometheus Node Exporter binaries. AIX and Solaris are not supported by any component in the stack. Remote monitoring must substitute for any hosts where agents cannot be deployed.

Zabbix Analysis — 01

Architecture Overview

Zabbix is a push/pull hybrid enterprise monitoring platform. A central Zabbix Server collects data from agents via active (agent-push) or passive (server-pull) modes. Regional Zabbix Proxies offload collection from the central server and buffer data locally — conceptually the closest equivalent to BMC's TSOM-era Integration Service tier, though Zabbix Proxy capacity is throughput-tunable rather than capped at a fixed agent ceiling. All components — server, database, proxy, agent, web UI — are Linux-native. No Windows components exist in the Zabbix infrastructure stack.

Central Layer — Primary Region / DC
  • Zabbix Server (2-node HA): Active/active or active/passive behind a VIP. All proxy and agent communication terminates here. Handles alert evaluation, action execution, and event generation.
  • Database Cluster (3-node PostgreSQL or MariaDB): All monitoring configuration (hosts, templates, items, triggers, macros), event history, and trend data stored here. Streaming replication with dedicated standby. This is the operationally most sensitive component.
  • Web/UI/API nodes (2-node, load-balanced): PHP-based frontend. Self-service portal integration target. Full REST API exposed for policy lifecycle management.
Regional Collection Layer — Per Region (UK, US, HK, MX)
  • Zabbix Proxies (active mode): UK: 6–8 proxies. HK: 5–7 proxies. US: 2–3 proxies. MX: 1–2 proxies. Proxies buffer data locally and push to the central server. Active mode works cleanly across firewalls.
  • Proxy sizing guidance: Zabbix documentation and community benchmarks suggest 3,000–10,000 NVPS (new values per second) per proxy depending on hardware. Under the confirmed model (60 baseline monitor instances/server; DB-hosted Linux 700–3600; 1-minute polling), total in-scope NVPS is ~43k baseline-only and ~128k–515k for DB-heavy mixed density. Proxy count must be sized to the actual DB-host distribution, not a flat host count.
  • Domain/environment split: Proxies should be split by domain (Unix OS, DB, application) and environment (prod/non-prod) to isolate collection failures and enable targeted patching.
Host Agent Layer — All 43k+ Managed Hosts
  • Zabbix Agent 2: Active mode. Reports to the regional proxy. Supports all item types: system metrics, log[], logrt[], log.count[], UserParameter, external scripts. Single agent binary covers OS, log, process, and script monitoring.
  • DB-specific monitoring: ODBC items or Zabbix DB templates using agent UserParameters for Oracle, PostgreSQL, MySQL. Or dedicated DB proxy with ODBC collection.
Policy and Governance Layer
  • Templates: Reusable monitoring configuration objects. A host inherits items, triggers, and graphs from attached templates. The baseline Linux template covers all standard OS monitoring — analogous to the BMC LINUX tag policy.

Zabbix Analysis — 03

Script Monitoring in Detail

Script monitoring is the area where Zabbix most closely replicates the BMC Patrol Scripting KM model, and where it most clearly outperforms Prometheus for this specific use case.

UserParameter — The Primary Pattern

A UserParameter is a custom item key defined in the Zabbix agent configuration (or injected via auto-registration). When Zabbix server requests the item, the agent executes the mapped command and returns its stdout as the item value. This is a direct analogue of the Patrol scripting KM execution model.

  • Configuration: UserParameter=custom.check.name[*],/path/to/script.sh "$1"
  • Script can return numeric values (for threshold evaluation) or text (for regex trigger evaluation)
  • Execution frequency is controlled by the item's update interval in Zabbix — directly analogous to Patrol KM polling interval
  • Trigger can evaluate the return value using any comparison operator or regex match
  • Per-host customisation via macros: the script path or arguments can use host-level macros for per-server variation

External Scripts

For scripts that must run server-side (not on the monitored host), Zabbix supports external script items. The Zabbix server executes the script on the server host and evaluates its output. Less common than UserParameters but available for cases where the script has no agent dependency.

Sizing Ceiling — Declared, Not Assumed
Sizing target is the full 60k Unix/Linux estate, not the 43k in-scope subset. The 300 BMC-unsupported-host surprise is the reason: architectural decisions must survive further in-scope expansion without redesign. Numbers below are upper bounds. Scaling beyond 60k (to 80k / 100k) adds proxies and DB nodes — it does not change the architecture.

Confirmed Data Volume Assumptions (60k Ceiling)

  • Planning ceiling: 60,000 servers.
  • Baseline NVPS (new values per second): ~60,000 — one metric per server-minute × 60 items.
  • DB-heavy NVPS: ~180,000–720,000. The upper edge drives the TimescaleDB design, not the average.
  • History retention: 90 days. Active TimescaleDB compression and aggressive trend management are mandatory, not optional.

Zabbix Scaling Model

Zabbix scales by adding proxies. Each proxy handles a subset of hosts, offloads collection from the central server, and buffers data locally (SQLite or PostgreSQL) when the central server is unreachable. This buffering property is the single most important fact for the regional-vs-central decision — it means Zabbix's "proxies regional + server central" topology is not a compromise, it is a first-class Zabbix architecture.

Zabbix Analysis — 07

Patching and Day-2 Operations

Agent Patching

Zabbix Agent 2 is a lightweight daemon. OS patch and reboot restarts it automatically via systemd. Agent binary upgrades are independent of OS patching and can be orchestrated via Ansible. The agent's UserParameter configuration is file-based — changes to agent-side scripts do not require an agent restart in most cases (depends on agent version). No KM versioning complexity comparable to BMC Patrol's KM compatibility matrix.

Proxy Patching

Proxies buffer data locally in SQLite or PostgreSQL. During a proxy restart for patching, agents continue to attempt connections; once the proxy is back, buffered data is forwarded. With 2+ proxies per domain/region, rolling proxy patching within the 15-day window maintains continuous collection coverage. Agent failover to a sibling proxy can be configured at the agent level for additional resilience.

Central Platform Patching

The Zabbix server HA pair allows rolling node patching. DB replication allows rolling DB node upgrades (replica first, then primary with failover). Zabbix web nodes are stateless — rolling restart is trivial. Total patching window for the central platform is approximately 4–8 hours with an HA design, well within a 15-day cycle's maintenance window.

Zabbix LTS Releases

Zabbix publishes Long-Term Support (LTS) releases with 5-year support windows. This aligns with change management — teams can standardise on an LTS version and receive security patches without major version upgrades for years. This is a meaningful operational advantage over Prometheus's faster release cadence.

Day-2 Operational Risks

  • Database size management: History and trend data accumulates. Without active partitioning and housekeeping configuration, the Zabbix DB will grow unbounded. Must be addressed in Day 1 design (TimescaleDB + retention policy).
  • UI performance at scale: The Zabbix web UI can become slow with large numbers of hosts and items if database queries are not optimised. Requires DB indexing and partition management.
  • Template version management: As templates evolve, hosts using older template versions must be updated. Without a formal template versioning discipline, template drift becomes a maintenance burden. Treat templates as code — version them in Git.
  • Trigger storm risk: A network event affecting many hosts simultaneously can generate thousands of triggers in seconds. Zabbix's event correlation (tag-based correlation and problem grouping in Zabbix 6.x) mitigates this but must be configured explicitly.

Prometheus on K8s — 01

Architecture Overview

Deployment Context — Confirmed
The target Kubernetes environment is a shared, managed cluster provided to our team as a namespace. We do not own the cluster. We do not install the control plane. We consume a namespace on an existing cluster, via a Kubernetes URL + credentials, and deploy our workloads as Helm charts or Kubernetes manifests. Everything below assumes that boundary.

Running Prometheus on Kubernetes is a different operational shape than running it on Unix VMs. On Unix, each Prometheus, Alertmanager, Grafana, and Loki runs as a systemd service on a dedicated RHEL VM. On K8s, each of those becomes a pod — an orchestrator-managed container — declared through a Helm chart. The components are the same; what changes is the delivery model, the failure model, and the ownership boundary.

The de facto installation pattern is the kube-prometheus-stack Helm chart (maintained by Prometheus-Community). It bundles:

  • Prometheus Operator — a controller that watches Kubernetes for Prometheus, ServiceMonitor, PrometheusRule, and Alertmanager custom resources, and reconciles the actual pods to match.
  • Prometheus server pods — actual scrape + storage engine, run as a StatefulSet with persistent volumes.
  • Alertmanager pods — alert routing, deduplication, silencing. HA-enabled by default.
  • Grafana — dashboard UI, Helm-integrated but optional.
  • Exporters — node-exporter (as DaemonSet for cluster nodes), kube-state-metrics, etc.
  • Loki (separate chart) — log stack; not part of kube-prometheus-stack by default. Added via its own Helm chart.
What K8s Changes About Prometheus
On K8s, Prometheus is declarative. You do not log into a server and edit prometheus.yml. You commit a ServiceMonitor or PrometheusRule YAML; the Prometheus Operator reconciles the running Prometheus config. This is a better governance model (everything is version-controlled, auditable, reversible) but it also means the team must be comfortable with Kubernetes as an operational surface — not just Prometheus.

Day-1 Installation Flow

  1. Cluster team provisions namespace monitoring with resource quota (CPU/memory ceiling, PVC quota).
  2. We configure kubectl / Helm to authenticate against the cluster URL.
  3. Pin chart version (e.g., kube-prometheus-stack-65.0.0). Never use latest.
  4. Create values.yaml — HA replica counts, retention, storage class, ingress.
  5. Deploy via helm install (or ArgoCD Application resource for GitOps).
  6. Wait for pods to reach Running + ready. StatefulSet ordering means Prometheus-0 before Prometheus-1.
  7. Validate ServiceMonitor/ PodMonitor discovery. Add first rule via PrometheusRule.
  8. Configure ingress to Grafana, Alertmanager, Prometheus UIs.
  9. Point external Alloy agents (on Unix hosts) at the Prometheus remote-write ingress.

Pods vs VMs — The Mental Model Shift
On Unix, a component = a VM. On K8s, a component = one or more pods, sharing the cluster's node pool. We do not see or manage the underlying nodes — the cluster team does. Prometheus on K8s — 06

Storage & Persistence

Three storage types are needed. Each has a different K8s shape.

  • Prometheus / VictoriaMetrics / Loki time-series storage: PersistentVolumeClaim (PVC) per pod via the cluster's StorageClass. At 90-day retention and 60k ceiling, total PVC footprint is ~28 TiB for metrics + logs. Confirm the StorageClass supports our IOPS demand (vmstorage and Prometheus are write-heavy; NVMe-backed or equivalent is expected).
  • Long-term object storage (optional but recommended): S3-compatible bucket (cluster's object storage, or external). VictoriaMetrics and Loki both support offloading old data to object storage. This reduces PVC usage and patches a cost dimension that PVC alone inflates.
  • Config / ephemeral: ConfigMap + Secret. Non-persistent.
PVC is a Lock-In Point
PVCs are bound to a specific cluster's StorageClass. Moving the platform to a new cluster is non-trivial — data must be exported/re-imported or snapshot-shipped. If there is any chance of cluster migration (cluster deprecation, re-provisioning), design for object storage backup from Day 1. This is the single biggest portability consideration with K8s-hosted monitoring.

The 60k Unix hosts are not migrating onto Kubernetes. They continue to run as they do today — RHEL VMs, systemd, Ansible-managed. What changes is where their metrics go: instead of a Prometheus service on a VM, they remote_write to a Prometheus/VictoriaMetrics endpoint exposed as a K8s Ingress or Service.

Ingress Becomes the Critical Path
On Unix-hosted Prometheus, the network path from agent to server is VM-to-VM on internal network. On K8s-hosted Prometheus, the path is agent → ingress controller → service → pod. Three new components in the path, each a possible failure mode. Ingress throughput (connections/sec, TLS CPU cost) must be sized for 60k concurrent agents pushing remote_write every 60 seconds — approximately 1000 req/sec sustained. Not huge, but non-trivial and must be validated.
Prometheus on K8s — 09

Day-1 POC Scope — What Must Be Validated

"POC from Day 1" in the K8s-IS-production model means the POC is not a lab exercise but a real production-track deployment in a non-production cluster (or a dedicated POC namespace with production-equivalent configuration). The POC exists to prove the architecture, not to explore it.

POC Success Criteria — Eight Items, Non-Negotiable

  1. Helm install of kube-prometheus-stack completes cleanly against the target namespace with pinned chart version and production values.yaml.
  2. 500 Unix hosts streaming via remote_write for 7 consecutive days with zero data loss. Alloy agent installed via Ansible.
  3. Ingress throughput validated at 10× POC scale (proxy 5000 hosts' worth of traffic through the ingress, verify no degradation).
  4. PrometheusRule and AlertmanagerConfig authored for 20 representative alerts (threshold, log-match, disconnect) — end-to-end to GMOM webhook.
  5. Pod restart resilience tested: kill Prometheus-0, kill vmstorage-3, verify data continuity and query consistency.
  6. Cluster upgrade rehearsal: the cluster team upgrades the target cluster by one minor version. Our namespace survives with zero intervention. If this fails, it is a programme risk, not a technical bug.
  7. PVC backup/restore validated: snapshot vmstorage PVC, restore to a new PVC, verify query correctness over the restored window.
  8. Quota exhaustion behaviour: intentionally exceed namespace CPU quota; verify pod eviction/pending behaviour matches expectation; document recovery playbook.
POC Duration & Exit Criteria
Plan 8–12 weeks for the POC. Exit criterion is not a technical checklist — it is operational confidence from the production support team that they can run this on Monday morning. If the POC succeeds technically but support cannot debug a pod in CrashLoopBackOff without escalation, the POC has not succeeded in the relevant sense.

Zabbix on K8s — 01

Architecture Overview

Deployment Context — Same as Prometheus on K8s
We consume a namespace on a shared cluster via Kubernetes URL. Helm charts deploy our workloads. The cluster team owns everything below the namespace boundary. This is a consistent K8s abstraction across both platforms; the architectural implications differ because Zabbix's components are stateful in ways Prometheus's are not.

Zabbix can run on Kubernetes, but it was not architected for it the way Prometheus was. The official Helm chart (zabbix/zabbix, maintained by Zabbix SIA) deploys the full stack as Kubernetes resources. It works. It is production-capable. It is not the native idiom — Zabbix's heritage is VM-hosted multi-tier architecture with a stateful database, not declarative, ephemeral, scheduler-managed pods.

The practical Zabbix-on-K8s deployment pattern:

  • Zabbix Server — StatefulSet, 2–3 replicas with Zabbix 7.0 native HA cluster.
  • Zabbix Proxy — Deployment or StatefulSet. Can run on K8s, but frequently better-placed on Unix VMs in regional DCs for WAN tolerance. Hybrid is acceptable and common.
  • PostgreSQL + TimescaleDB — StatefulSet with PVCs, or external database (recommended at 60k scale).
  • Zabbix Web frontend (nginx + PHP) — Deployment, stateless, horizontally scalable.
  • Zabbix Agent — not on K8s. Runs on Unix hosts as today.
Key Architectural Decision for Zabbix on K8s
Where does TimescaleDB live? In-cluster (StatefulSet + PVC) or external (dedicated DB VMs or managed DBaaS)? At 60k ceiling with 180k–720k NVPS, the database is the critical resource. See the dedicated section below.

This is the single most consequential decision for Zabbix on K8s at 60k ceiling. It is essentially the same database sizing conversation as Zabbix on Unix — the question is where the database lives relative to the K8s cluster.

OptionDescriptionProsCons
A. In-cluster (StatefulSet + PVC) TimescaleDB runs as pods in the same K8s namespace via an operator (CloudNativePG, Zalando Postgres Operator, or Crunchy). Single deployment model. Recommendation
Option B (External dedicated DB VMs) is the lowest-risk choice at 60k ceiling for an organisation new to K8s. The DB is already the critical component in any Zabbix deployment. Keeping it on known infrastructure (Unix VMs, known DBA practices, known backup tooling) while Zabbix Server benefits from K8s deployment is a clean separation of complexity. Option A becomes viable once the team has 18+ months of K8s operations maturity and an operator-based PostgreSQL is validated at this scale.
Zabbix on K8s — 06

Proxies on K8s vs Unix — A Real Choice

Zabbix proxies can run on K8s or on Unix VMs. The decision is not binary; it can be per-region. The variables:

  • Cluster availability per region. If UK is the only K8s cluster, UK proxies can run in the cluster. HK/US/MX proxies must run on Unix VMs (or on regional K8s clusters if provided by cluster team).
  • WAN buffering. Proxies are precisely the component that benefits from regional placement close to the agents. A proxy in the UK K8s cluster serving HK hosts gives you no WAN tolerance at all — it's just a proxy on the wrong side of the WAN.
  • Operational consistency. Running proxies on Unix in HK/US/MX while UK proxies run on K8s is a two-flavor ops model. Acceptable but must be documented.
Pragmatic Recommendation for Proxies
UK proxies in K8s; HK/US/MX proxies on Unix VMs. This places the WAN-sensitive proxies close to their agents, keeps the regional DC footprint minimal, and avoids requiring a K8s cluster in every region. As K8s matures in other regions, proxies can migrate in-cluster per region. This is architecturally clean.
Zabbix on K8s — 07

Cluster Ownership Boundary

The ownership model is structurally identical to Prometheus-on-K8s (see that tab for full detail). The functional differences for Zabbix:

  • External DB adds a DBA boundary inside our team. Platform team runs Zabbix Server pods; DBA runs TimescaleDB on Unix. Clear handoff point.
  • Hybrid proxy model means Unix ownership continues for HK/US/MX proxies. Ansible, systemd, RHEL patching — all unchanged.
  • Agent ownership unchanged — 60k Unix hosts continue under Ansible management. K8s does not touch agent delivery.
Net Assessment
Zabbix on K8s is a partial K8s adoption. Server and Web go to K8s; DB, regional proxies, and agents remain on Unix. This is actually the lower-risk path into K8s — it surfaces K8s operational gaps against components that are least stateful, while keeping the stateful critical path (DB) on mature infrastructure.

Management Brief — What "Deploy on K8s" Actually Means

Audience
This section is written for management readers with no Kubernetes background. It is a conceptual primer, not a technical reference. Skip if familiar.

Kubernetes in One Paragraph

Kubernetes (K8s) is a cluster-level operating system for containerised applications. Where today we provision a RHEL VM, install software on it, configure services, and monitor it — in Kubernetes, we declare what we want (e.g., "I want 3 copies of the Zabbix Server container running, with 8 CPU and 24 GB RAM each, backed by 500 GB of storage, reachable at this URL"), hand that declaration to the cluster, and the cluster finds where to run the containers, restarts them if they fail, moves them if a server goes down, and keeps the declared state. We describe intent; the cluster enforces it.

The Unit Model

ConceptPlain-English Equivalent
ContainerA packaged application with everything it needs to run — like a portable app installer that runs identically anywhere.
PodOne or more containers running together as a single unit. In most cases, one pod = one application instance.
NodeA physical or virtual server that hosts pods. We do not manage nodes — the cluster team does.
NamespaceA logical partition of the cluster. Our team gets a namespace and everything we deploy lives inside it. Think of it as "our reserved space" on a shared building.
Deployment / StatefulSetA recipe telling the cluster "keep N copies of this container running". Deployment for stateless; StatefulSet for stateful (like databases).
ServiceA stable network endpoint in front of pods — pods can come and go; the Service stays reachable.
IngressThe front door to our Services from outside the cluster — where URLs are served.
PersistentVolumeClaim (PVC)A request for durable storage attached to a pod. Survives pod restarts. This is where databases keep their data.
Helm chartA pre-packaged deployment template — like an Ansible role, but for K8s. We give it a values.yaml with our settings; it generates and applies the full deployment.
ConfigMap / SecretConfiguration and sensitive data injected into pods. Equivalent to /etc/config files, but managed by the cluster.

Management Brief — End-to-End Deployment Flow

The mechanical steps to stand up Prometheus or Zabbix on our K8s environment:

  1. Cluster access provisioned. The cluster platform team creates a namespace for us (e.g., monitoring-prom) and gives the platform team a kubeconfig file — credentials to authenticate against the Kubernetes URL. This is a one-time request.
  2. Resource quota agreed. The cluster team caps how much CPU, memory, and storage our namespace may consume. We declare our requirements up-front (see sizing sections). Under-quota = silent failure later. Over-quota = cluster team pushback up-front. The honest conversation happens at namespace creation.
  3. Values.yaml authored and version-controlled. The platform team writes our deployment configuration — replica counts, retention, resource requests, ingress hostnames, storage class, Alertmanager routing. This file lives in Git, reviewed as code.
  4. Chart deployed. One command — helm install monitoring prometheus-community/kube-prometheus-stack -n monitoring -f values.yaml — and the cluster starts creating pods. Within 5–10 minutes, the full monitoring stack is running.
  5. Verification. We run kubectl get pods -n monitoring; every pod should be Running and Ready. If a pod is CrashLoopBackOff, the values or infrastructure is wrong — diagnose via kubectl logs and kubectl describe.
  6. Exposure. Ingress resources publish Grafana, Alertmanager, and the remote-write endpoint to URLs — e.g., grafana.monitoring.internal. Unix agents point at the remote-write URL; people browse to the Grafana URL.
  7. Rules / templates / agents. From here onward the work is the same as Unix: author PrometheusRules or Zabbix templates, roll agents via Ansible, validate alerts flow end-to-end to GMOM.
The Fundamental Simplification
Today: "Provision 30 VMs, install RHEL, install packages, configure services, configure systemd, monitor the VMs, patch the VMs, handle failure."
On K8s: "Write a values.yaml, run helm install, watch pods come up." The tradeoff is that everything below the namespace line — the VMs, the OS, the cluster itself — becomes someone else's system. When it fails, we file a ticket; we do not fix it.

Recommendation Summary

Recommended Ordering Under the Constraints Given

Given: BMC-trained team with 0 K8s exposure, 60k estate ceiling, portal Day-1 requirement, operational risk reduction as primary decision lens, and management direction that K8s is Day-1 production.

  1. First choice: Zabbix on K8s (hybrid). Zabbix Server + Web on UK K8s cluster; external TimescaleDB on Unix VMs; UK proxies on K8s; HK/US/MX proxies on Unix; agents on Unix. This is the path that respects both the K8s Day-1 directive AND the BMC-to-Zabbix operational transfer that makes the BAU team productive in weeks, not months. Best of both mandates.
  2. Second choice: Zabbix on Unix (Model B — UK-central + regional proxies). If K8s cluster availability, stability, or cluster-team readiness is not adequate by programme start, fall back to this. The migration to K8s remains possible in 18–30 months without re-architecting monitoring logic.
  3. Third choice: Prometheus on K8s. Strategically the strongest fit with a K8s future, but imposes the highest compound learning curve on a team with no K8s exposure AND no Prometheus exposure. Viable only with (a) dedicated 8–12 week K8s training up-front, (b) expanded platform team (4+ engineers), (c) extended portal Day-1 timeline or GitOps-acceptable interim.
  4. Fourth: Prometheus on Unix. Combines the highest operational friction (Prometheus concepts unfamiliar to team) with the lowest strategic upside (not aligned with K8s future). Recommended against under the current constraint set.

Critical caveat for the first-choice recommendation: Zabbix on K8s is only first-choice if the cluster is stable, the cluster team is responsive, and we can get named liaison + quota contract agreement before Day 1. If any of those is uncertain, fall back immediately to Zabbix on Unix. Do not start on K8s with ambiguity about the cluster boundary.

Architect-Level Framing
Every option in the matrix is technically feasible at 60k. The decision is not "can it scale" — it is "can the team operate it with the risk budget we have". The programme will not be judged on the footprint count in the sizing tables; it will be judged on whether the BAU team can answer a page at 3 a.m. without escalation in Month 12. Weight the recommendations through that lens.

Delivery — 05

15-Day Patch Cycle — Operational Plan

Agent-Side (43k Hosts)

Zabbix Agent 2 is a lightweight daemon. OS patch and reboot is handled by systemd restart — no manual intervention required at scale. Agent binary version upgrades are orchestrated via Ansible as a separate, lower-frequency activity (typically aligned with Zabbix LTS release upgrades, not every 15-day cycle). Agent configuration changes (UserParameter additions) are deployed by Ansible alongside the policy migration programme — they do not affect OS patch scheduling.

Proxy Tier (15–21 proxies)

Proxies buffer data locally during downtime. With 2+ proxies per domain/region, rolling patching maintains continuous collection:

  1. Take Proxy A offline for patch. Proxy B continues collecting from its agent subset.
  2. If agent sets are split between proxies, agents targeting Proxy A begin buffering locally (limited buffer) or fail over to Proxy B via failover configuration.
  3. Proxy A patched, validated, returned to service. Proceed to Proxy B.

At region-weighted proxy density (UK/HK higher, US/MX lower), rolling through all proxies in a 15-day cycle is achievable with daily patch windows.

Central Platform (5 nodes)

Server HA pair: patch secondary, validate, flip VIP, patch primary. DB cluster: patch replicas first, promote one replica, patch original primary as new replica. Web nodes: patch one at a time behind load balancer. Total planned downtime for central platform: 2–4 hours per 15-day cycle, all within maintenance window.

Decision Rationale — Architecture Approval Summary

Recommended Platform: Zabbix

Against the six confirmed decision parameters — operational risk reduction as the primary lens, shared prod support + platform team ownership, BMC-trained-only team, portal required Day 1, AIX/Solaris phasing out, Kubernetes uncertain — Zabbix is the recommended replacement platform for the Unix/Linux and database monitoring estate.

Six Arguments for Architecture Approval

  1. Patrol-parity functionality is native, not assembled. Every critical Patrol KM capability — script monitoring (UserParameter), log monitoring (log[]/logrt[]), agent disconnect (host availability), per-host threshold customisation (macros), ICMP ping (built-in) — is a single-component, native Zabbix capability. Prometheus achieves the same outcomes through a composite pipeline. In a production support operating model, composite pipelines are failure points. Native is lower risk.
  2. The shared ownership model maps cleanly to Zabbix's architecture. Production support operates the Zabbix console and portal for BAU — concept model transfers from BMC in 3–4 weeks. The platform team owns DB, proxy architecture, template versioning, and upgrades — all well-defined and separable responsibilities. No role blurring. No escalation ambiguity.
  3. Portal Day 1 is achievable on Zabbix's API, not on Prometheus's. The Zabbix API object model (host/item/trigger/macro) is functionally analogous to the BMC API model the portal already calls. The portal backend is a swap with UI changes, not a rebuild of the logic layer. With 2 portal developers in Phase 1, a full-capability portal is deliverable before UK cutover. On Prometheus, this timeline is high-risk given the YAML orchestration and multi-component abstraction required.
  4. AIX/Solaris being phased out removes a historical Zabbix gap. In environments where AIX required monitoring, Zabbix agent 2's lack of AIX support was a complication. With AIX/Solaris phasing out and explicitly out of scope, this is a non-issue for this programme. Existing Patrol coverage on those hosts continues to decommission.
  5. Scale is achievable with the right architecture — and the risk is engineering, not platform. Zabbix at 60k hosts is documented. The risk is the database tier — TimescaleDB is mandatory, DBA involvement is mandatory, NVPS budgeting is mandatory. These are quantifiable, costed engineering decisions, not architectural unknowns. The platform team handles this. It is not a production support concern.
  6. Kubernetes uncertainty eliminates Prometheus's primary strategic argument for this programme. Prometheus's strongest advantage is K8s-native observability. With management not having made a K8s decision, that advantage is not a current programme deliverable. Choosing Prometheus for a K8s future that may be years away — at higher portal risk and higher BAU operator friction — inverts the operational risk reduction lens that governs this decision.
Final Decision Recommendation

Proceed with Zabbix as the primary monitoring platform for the Unix/Linux and database estate replacement programme.
