Section 01

Executive Summary

Primary Recommendation
Prometheus + Alertmanager + Grafana (PAG Stack) — deployed with a tiered, federated architecture — is the recommended open-source foundation for replacing BMC TrueSight / Helix at this scale. This is not a single-product swap. It is a platform composition that satisfies functional requirements, is deployable on Unix/Linux only, has a mature commercial support ecosystem, and extends cleanly to Kubernetes.

No single open-source product offers a direct one-to-one functional replacement for BMC Helix Operations Management across 60,000 Unix/Linux servers with 38,000 custom policies. The realistic replacement is a curated stack, not a product.

The recommended stack is:

  • Prometheus — core metrics collection, threshold evaluation, and alerting rules engine
  • Grafana Alloy (formerly Grafana Agent) — lightweight agent for log and metric forwarding on each host
  • Loki — structured log aggregation replacing Patrol log KM functionality
  • Alertmanager — alert deduplication, grouping, silencing, and GMOM/ITSM routing
  • Grafana — unified observability frontend, dashboarding, and on-call routing
  • Blackbox Exporter — ICMP/ping and protocol-level reachability monitoring

Secondary components required for production-grade operation at this scale:

  • Thanos or Cortex — for long-term metric retention and multi-region query federation
  • GitOps layer (Git + CI pipeline) — for versioned policy governance replacing informal change tracking
  • Custom Exporter framework — for the ~700 Patrol scripting KM equivalents
Architectural Caution
Any organisation expecting to lift-and-shift 38,000 policies from BMC Patrol into Prometheus without a structured migration programme will fail. The complexity is real, the mapping is not one-to-one, and the operational model will need to change. This document addresses each of those dimensions explicitly.

What This Delivers

  • A fully Unix/Linux-native monitoring platform with no Windows dependency
  • A federated, multi-region architecture that scales past the 60k server ceiling
  • Commercial support available via Grafana Labs enterprise licensing
  • A clear and unambiguous path to Kubernetes monitoring without introducing a second platform
  • Policy-as-code via Prometheus recording rules and alerting rules stored in Git
  • Event forwarding to existing GMOM/ITSM infrastructure via Alertmanager webhook routing
Section 02

Open-Source Candidate Assessment

The following candidates were evaluated against the bank's stated requirements. Each is assessed on its own merits before the stack recommendation is made.

Prometheus + Grafana Ecosystem

Architecture: Pull-based metrics collection. Prometheus scrapes exporters on each target. Alertmanager routes alert notifications. Grafana provides the observability frontend. Loki handles log aggregation. Alloy (formerly the Agent) provides a unified on-host collection component.

Strengths at this scale: The most widely deployed open-source metrics platform globally. Mature federation and sharding support. The Grafana Labs commercial offering (Grafana Enterprise, Grafana Cloud) provides support contracts, training, and SLA-backed services. Kubernetes-native via kube-state-metrics and the Prometheus Operator. Alertmanager is purpose-built for enterprise alert routing.

Realistic gaps: No native agent disconnect detection — requires custom dead man's switch alerting pattern. Script monitoring requires custom exporter wrappers. No built-in self-service portal — requires separate tooling or custom build. Metrics cardinality at 60k servers requires careful sharding design.

Datadog Open-Source Components (Excluded)

Datadog is a commercial SaaS platform. Its open-source agent is present but the platform itself is not open-source. Excluded from consideration.

Zabbix

Architecture: Push and pull hybrid. Centralised Zabbix Server/Proxy architecture. Active and passive agents. Built-in database backend (MySQL/PostgreSQL). Native UI for host management, templates, and alerting.

Strengths at this scale: Closer BMC-style operational model — template-based monitoring, agent-level config management, and a built-in web interface for operations teams. Agent disconnect detection is native. Script-based monitoring is a first-class citizen via Zabbix external scripts and UserParameters. Commercial support available through Zabbix LLC.

Realistic gaps: Scaling beyond 20–30k actively monitored hosts requires significant proxy architecture investment. The Zabbix database becomes a bottleneck at high polling frequency and scale. Kubernetes integration is achievable but architecturally awkward — Zabbix was not built for ephemeral container workloads. UI is functional but does not offer modern observability experience. GitOps-native config management is an afterthought, not a design principle.

Nagios / Icinga2

Architecture: Check-based monitoring. A scheduler dispatches checks to active agents or via NRPE/SSH. Results are processed and alerts fired on state transitions. Icinga2 is the modernised fork with native clustering.

Strengths: Conceptually the closest model to Patrol-style check execution. Strong community and plugin ecosystem. Icinga2's Director module provides a database-backed configuration layer with a web interface for self-service-style operations.

Realistic gaps: At 60,000 Unix/Linux servers, Nagios/Icinga2 check scheduling at 1–5 minute intervals will create scheduler overload without extreme tuning and horizontal cluster distribution. Not designed for 38,000 individual check configurations at this density. Kubernetes monitoring support is minimal and not fit for a unified platform strategy. Commercial support for Icinga2 (NETWAYS/Icinga GmbH) exists but is not enterprise-grade at banking scale.

Victoria Metrics

Architecture: A high-performance, Prometheus-compatible time-series database and metrics aggregation engine. Not a full monitoring platform — purely a storage and query layer.

Role in the stack: VictoriaMetrics is an important consideration as a Prometheus storage backend replacement in high-cardinality, high-ingest environments. At 60k servers, VictoriaMetrics outperforms Prometheus's native TSDB on storage efficiency and query performance, and offers better cluster mode stability than Thanos for some topologies. It does not replace the full stack but can replace Prometheus's storage tier.

OpenTelemetry

Architecture: Vendor-neutral telemetry collection framework. Not a monitoring platform — a data pipeline standard. The OpenTelemetry Collector can ingest metrics, logs, and traces and forward to multiple backends.

Role in the stack: OpenTelemetry Collector is worth introducing as a collection normalisation layer, particularly for database monitoring and future application monitoring. It does not replace Prometheus or Loki as backends but provides pipeline flexibility.

Elastic Stack (ELK/EFK)

Architecture: Elasticsearch as storage, Logstash/Fluentd/Filebeat for collection, Kibana for visualisation. Elastic now ships a unified observability product (Elastic Observability).

Strengths: Log monitoring is best-in-class. Full-text search across billions of log events is a genuine capability. Elastic Observability now includes metrics, traces, and uptime monitoring under one platform.

Realistic gaps: Elasticsearch at 60k server scale requires substantial hardware investment and operational expertise. Kibana-based monitoring does not replicate the Patrol policy model. Alerting is less mature than Alertmanager. Elastic's open-source licensing (SSPL from v7.11 onwards) introduces licence risk in a financial services context. Commercial Elastic support is available but expensive at scale.

Assessment Outcome
The Prometheus + Grafana ecosystem is the only candidate with a realistic path to covering all five review criteria at this scale while maintaining a single logical platform into Kubernetes. Zabbix is a credible alternative for organisations with lower Kubernetes priority. Neither Nagios/Icinga2 nor pure ELK is architecturally appropriate as the primary platform at 60k+ server scale in 2025 onwards.
Section 03

Stack Recommendation

Recommended Stack: PAG-Extended

The recommended target state is a composed stack across three logical tiers:

Tier 1 — Collection (Per Host)

  • Grafana Alloy — unified agent deployed on every managed Unix/Linux host. Replaces Patrol KM agents for log, metrics, and custom script output collection. Supports scraping local exporters, tailing log files, and executing scripts with output forwarding.
  • Node Exporter — OS-level metrics (CPU, memory, disk, filesystem, kernel). Covers baseline LINUX-tag policy equivalent.
  • Custom Exporter framework — thin shell-wrapper pattern to expose script output as Prometheus metrics (text file collector or HTTP exporter). Required for Patrol scripting KM equivalents.
  • Database Exporters — postgres_exporter, oracledb_exporter, mysqld_exporter for the 10k+ DB instance coverage.
  • Blackbox Exporter — ICMP ping monitoring. Deployed per-region, not per-host.

Tier 2 — Aggregation and Storage (Per Region)

  • Prometheus (sharded) — regional scraping clusters. Federation or sharding via consistent hash or functional partitioning (OS vs DB vs application). Target: 3–5 Prometheus instances per region at this scale.
  • Loki — regional log aggregation. Receives structured logs forwarded by Alloy agents. Provides log-based alerting equivalent to Patrol log KM.
  • Thanos (or VictoriaMetrics Cluster) — long-term retention, cross-region query federation, and compaction. Decision between Thanos and VictoriaMetrics should be made based on operational team capability. VictoriaMetrics has lower operational complexity; Thanos has broader enterprise adoption.
  • Alertmanager — per-region with mesh configuration for HA. Handles deduplication, grouping, silencing, and outbound routing to GMOM/ServiceNow webhooks.

Tier 3 — Presentation and Governance (Global)

  • Grafana Enterprise — global unified dashboarding, on-call management (Grafana OnCall), alert state visualisation. RBAC for self-service portal equivalent.
  • Git repository (GitOps) — alerting rules, recording rules, and scrape configurations stored as code. CI pipeline (Jenkins/GitLab CI) validates and deploys config changes. This is the version control layer that current BMC change tracking does not provide.
  • Ansible — exporter deployment, Alloy agent configuration, and OS-level monitoring configuration at 60k scale. Integrates with existing Ansible capability.
Section 04

Target-State High-Level Architecture

The following represents the recommended infrastructure topology. All components are Unix/Linux only. No Windows components are present in this architecture.

Layer 0 — Managed Hosts (60k Unix/Linux + 8k DB servers)

Grafana Alloy agent — lightweight, deployed via Ansible on all managed hosts. Responsible for: log tail forwarding to Loki, local exporter scrape relay, script execution wrapper invocation.

Node Exporter — OS baseline metrics, exposed on port 9100 locally. Scraped by regional Prometheus.

Custom Script Exporters — thin Python/shell wrappers using Prometheus text file collector (/var/lib/node_exporter/textfile). Covers the ~700 Patrol scripting KM equivalents and ~several hundred shell log KM equivalents.

DB Exporters — Oracle, PostgreSQL, MySQL exporters on DB hosts, scraped by dedicated DB Prometheus shard.

Layer 1 — Regional Aggregation (Per Region: UK, US, HK, MX)

Prometheus Cluster (3–5 nodes, sharded by function) — Shard 1: OS/infrastructure metrics (Node Exporter). Shard 2: Database metrics. Shard 3: Custom exporters and script metrics. Each shard runs 2 Prometheus instances in active-active for HA, scraping the same targets (Prometheus does not have native clustering — HA via duplication).

Loki Cluster — Distributed log ingestion. Grafana Alloy on each host ships logs to regional Loki. Log-based alerting rules replace Patrol log KM alert conditions.

Alertmanager (2-node mesh) — Active-active HA. Routes to: outbound GMOM webhook, ServiceNow incident API, email, and PagerDuty/Grafana OnCall.

Blackbox Exporter — Regional deployment, ICMP monitoring of all in-region targets. Agent disconnect detection via absence alerting (up == 0 or Alloy heartbeat absent).

Layer 2 — Global Federation and Long-Term Storage

Thanos Query / VictoriaMetrics VMselect — Global query layer. Enables cross-region dashboard queries in Grafana. Aggregates from regional Prometheus or VictoriaMetrics nodes.

Object Storage (S3-compatible, on-prem) — Thanos or VictoriaMetrics long-term block storage. Ceph or MinIO on-prem if cloud object storage is not available. Enables 12–24 month metric retention without local disk growth.

Loki Compactor + Object Storage — Log retention management with configurable per-label retention policies.

Layer 3 — Operations and Governance

Grafana Enterprise (global) — Unified UI across all regions. Role-based access for production support teams, onboarding teams, and application self-service. Dashboards provisioned via code (Grafonnet or dashboard JSON in Git).

Git (Policy-as-Code) — Prometheus alerting rules, recording rules, scrape_configs, Loki alerting rules, and Alloy configs all stored in version-controlled repositories. Change review via PR/MR process. Replaces informal BMC change tracking with auditable GitOps workflow.

Ansible (Config Delivery) — Exporter installation, Alloy agent configuration, alerting rule deployment. Leverages existing Ansible estate.

Alertmanager → GMOM → ServiceNow — Event forwarding preserved. Alertmanager webhooks send enriched JSON payloads to the existing GMOM/event enrichment layer. GMOM continues to handle suppression, enrichment, and ServiceNow incident creation.

HA Design Note
Prometheus does not have native clustering. High availability at this scale is achieved by running two Prometheus instances per shard scraping identical targets. Alertmanager handles deduplication via mesh. For simplified HA with native clustering, VictoriaMetrics Cluster mode is an alternative that removes the duplication requirement.
Section 05

Custom Monitoring Depth: Scripts via Log Components

This is the section that most architecture reviews understate. At this bank, approximately 700 servers run custom monitoring via Patrol scripting KM, and several hundred more use shell-script monitoring through log KM. This is not a minor edge case — it is core monitoring logic.

The BMC Patrol Model (Current State)

In Patrol, a script KM executes a script, captures its output or exit code, and compares it against a threshold or regex pattern. The log KM tails a log file and fires an alert when output matches a rule. Both models treat text output as a first-class monitoring signal.

The Prometheus/Loki Equivalent

Script Monitoring → Text File Collector

Prometheus's text file collector allows any process to write a .prom file in a designated directory (/var/lib/node_exporter/textfile/). Node Exporter reads these files and exposes their contents as standard Prometheus metrics. This means any shell script can produce monitoring data by writing a file.

A standard wrapper pattern:

  • Existing Patrol script runs via cron or systemd timer
  • Script output is converted to Prometheus text format by a thin wrapper
  • Node Exporter exposes the resulting metric
  • Prometheus scrapes it, evaluates threshold rules, fires alerts via Alertmanager

This pattern supports both numeric output (gauge/counter metrics) and text output (info metrics with label encoding). The wrapper is 10–20 lines of shell. Migration of 700 scripting KM equivalents is a structured programme of work, not a technical impossibility.

Log Monitoring → Loki Alerting Rules

Loki supports alerting rules using LogQL — the log query language. A Patrol log KM rule like "alert if this string appears in /var/log/app.log" translates to a Loki alerting rule using a log stream selector and a line filter expression. Rules are stored as YAML, managed in Git, deployed via CI.

Grafana Alloy on each host handles log tailing, label injection (hostname, environment, policy tag), and forwarding to Loki. The Patrol log KM's per-server customisation is replicated via Alloy configuration scoped per-host or per-group using Alloy's config block model.

What Cannot Be Directly Replicated

Complexity — Do Not Understate
  • Scripts that perform complex multi-step logic with internal state will require refactoring, not just wrapping. Patrol KM scripts that interact with the Patrol agent API directly are not portable.
  • Scripts that return structured multi-value output need a more complex exporter wrapper — not a simple text file write.
  • Real-time log alerting in Patrol (sub-minute) is achievable in Loki but requires correct Alloy pipeline and Loki ruler configuration. Default Loki ruler evaluation interval is 1 minute — this must be confirmed acceptable or reduced at cost of load.
  • The 38,000 custom policies cannot be migrated automatically. A policy audit, classification, and migration programme is required. Estimate: 12–18 months for full migration with a dedicated team.

Script Exporter Alternative

For complex scripts, the Prometheus script_exporter (open-source, third-party) allows Prometheus to execute scripts at scrape time and parse their output as metrics. This is closer to the Patrol scripting KM execution model. It introduces a synchronous script execution risk at scale and should be used selectively — not as a default pattern for all 700 scripts.

Section 07

Patching, Platform Support, and Day-2 Operational Impact

15-Day Patching Cycle

The bank's 15-day patching cycle applies to all managed infrastructure. This has direct implications for monitoring platform components.

Agent-Side (Alloy / Node Exporter)

Grafana Alloy and Node Exporter are static binaries. They have no runtime dependencies and do not require OS package management for operation. Patching the OS underneath them does not require agent reconfiguration in most cases. Agent restart after OS patch is handled by systemd service configuration — no manual intervention required at scale.

Alloy itself will have release cadence updates. Agent binary updates can be orchestrated via Ansible at scale within the 15-day window. This is significantly simpler than Patrol KM patching, which requires coordination with the Patrol server and knowledge module version compatibility checks.

Platform-Side (Prometheus, Loki, Alertmanager)

These are long-running server processes. A HA design (active-active pairs) allows rolling upgrades during the patch window without monitoring gaps. Prometheus rolling restart during scrape cycle results in at most one scrape interval gap (typically 60 seconds) — acceptable for operational monitoring.

Loki and Thanos/VictoriaMetrics use object storage backends. Component restarts do not lose data. Rolling upgrades are supported by all three projects.

OS-Level Compatibility

All stack components are distributed as Linux binaries or container images. They run on RHEL 7+, Oracle Linux 7+, Ubuntu, and Debian. AIX and Solaris are explicitly not supported by any component in this stack. For AIX and Solaris hosts, monitoring must be addressed via remote black-box checking (SNMP, remote script execution, or SSH-based checks via Blackbox Exporter extensions). This is a genuine coverage gap that must be acknowledged in the migration programme.

AIX and Solaris — Coverage Gap
No component in the Prometheus/Grafana stack runs natively on AIX or Solaris. Prometheus exporters are Linux/macOS binaries. If AIX and Solaris hosts require in-process agent monitoring (equivalent to Patrol agent), a separate solution or remote monitoring strategy must be defined. Options: remote SSH-based checks via Prometheus blackbox+script, SNMP polling, or Zabbix agent (which does compile for AIX). This is not a blocker but must be scoped explicitly in Phase 1.

Day-2 Operations Burden

Configuration Drift

BMC Patrol's operational model centralises configuration on the server. The PAG stack's GitOps model distributes config across Git, CI, Alloy agent configs, and Prometheus rule files. Without discipline, this introduces configuration drift risk. The mitigation is mandatory GitOps — all configuration changes go through Git, not ad-hoc file edits on monitoring servers. This requires a cultural shift for production support teams trained on BMC's centralised model.

Prometheus Cardinality Management

At 60k servers, cardinality explosion is a real operational risk. Poorly designed custom exporters or log label schemas can cause Prometheus memory exhaustion. Production support teams must be trained in cardinality analysis. Grafana Enterprise includes cardinality management tools. This is a day-2 discipline requirement, not a day-0 architecture failure.

Alert Noise and Runbook Maintenance

Migrating 38,000 policies to Alertmanager rules will initially produce significantly more alert noise than the tuned BMC environment. A noise reduction programme running in parallel with migration is required. This is expected and should be planned as a named workstream, not an afterthought.

On-Call Model

Grafana OnCall (part of Grafana Enterprise) provides on-call schedule management, escalation policies, and alert routing. This directly replaces any BMC on-call routing currently handled through GMOM. Integration with ServiceNow remains through the existing webhook path.

Section 08

Kubernetes and Container Monitoring — Future-State Strategy

The PAG stack is the de facto standard for Kubernetes observability. This is not a qualified statement — it is the operational reality across the industry.

Kubernetes Monitoring via the Same Platform

When Kubernetes adoption begins, the following components are added to the existing PAG stack without architectural disruption:

  • Prometheus Operator — manages Prometheus, Alertmanager, and ServiceMonitor CRDs inside Kubernetes clusters. Kubernetes workloads define their own monitoring via ServiceMonitor resources, integrating with the same Alertmanager and Grafana.
  • kube-state-metrics — exposes Kubernetes API object state as Prometheus metrics (pod status, deployment replicas, node conditions).
  • Grafana Alloy on Kubernetes — deployed as a DaemonSet. Handles container log collection and forwarding to the same Loki cluster used for Unix/Linux server logs.
  • Thanos / VictoriaMetrics federation — Kubernetes cluster Prometheus instances federate to the same global query layer as regional Unix monitoring Prometheus instances. Single Grafana query endpoint for operators.

Single Logical Platform

This is the primary architectural advantage of the PAG stack over any alternative. A production support engineer monitoring a RHEL 8 server and a Kubernetes pod uses the same Grafana interface, the same Alertmanager routing, the same on-call tool, and the same alert rule syntax. This is not achievable with Zabbix + a separate Kubernetes stack, or Icinga2 + Prometheus.

Long-Term Platform Continuity
The risk of a monitoring platform split — where Unix infrastructure uses one tool and Kubernetes uses another — is a significant operational and cost liability. The PAG stack eliminates this risk by design. It is the single strongest argument for choosing this stack over Zabbix even if Zabbix is operationally preferable for the legacy Unix estate alone.

Section 11

Decision Rationale — Architecture Approval

This section is intended for Architecture Review Board and leadership sign-off. It compresses the preceding analysis into the five arguments that constitute the recommendation.

1. Scale is Non-Negotiable

The Prometheus ecosystem has documented, reference deployments at 60,000+ server scale with federation and sharding. No other open-source candidate can make this claim with comparable production evidence. At the bank's scale, choosing a platform without a proven ceiling is not a risk worth taking.

2. The Kubernetes Problem Must Be Solved Once

When the bank's infrastructure moves toward containers, the monitoring platform decision becomes a 10-year decision, not a 3-year one. The PAG stack is the only option that makes the Kubernetes transition a configuration addition rather than a platform replacement. The cost of running two monitoring platforms — one for legacy Unix and one for Kubernetes — exceeds the cost of migrating to a single platform now.

3. Commercial Support Is Available and Mature

Grafana Labs offers enterprise support contracts, training, and SLA-backed services. This is not an unsupported open-source project. The financial services sector has precedent for Grafana Enterprise deployments at banking scale. This de-risks the "who do we call at 3am" concern that production support leadership will raise.

4. Custom Monitoring Is Solvable, Not Trivial

The 700+ Patrol scripting KM equivalents and 38k custom policies are not a technical blocker. They are a structured migration programme. The text file collector and Loki alerting rule patterns cover the vast majority of cases. What requires genuine engineering effort is the audit, classification, and wrapper development — not a fundamental architectural limitation. This work should be scoped as a named programme with dedicated resourcing, not as part of the platform team's background work.

5. GitOps Governance Improves the Current State

The current BMC environment has change tracking but not formal versioning. The PAG stack's GitOps model produces an audit trail, rollback capability, and peer-reviewed policy changes as a structural byproduct of normal operations. This is an improvement in governance posture, not just a technology change. For a banking environment, this argument resonates at the CISO and risk function level, not just engineering.

Final Recommendation

Adopt the Prometheus + Grafana + Loki + Alertmanager stack as the target-state monitoring platform for the bank's Unix/Linux infrastructure estate.

Proceed to Phase 1 (Foundation) with:

  • UK region as the primary migration zone
  • Grafana Labs Enterprise engagement for commercial support and training
  • Thanos vs VictoriaMetrics decision within Phase 1 based on team capability assessment
  • AIX/Solaris remote monitoring strategy defined before Phase 2 commencement
  • Dedicated migration team staffed — minimum 4 engineers plus a migration programme lead
  • Policy audit workstream initiated in parallel with Phase 1 technical deployment

Estimated migration timeline: 20–24 months to full BMC decommission. Dual-platform cost window: 12–18 months depending on regional decommission sequencing.
